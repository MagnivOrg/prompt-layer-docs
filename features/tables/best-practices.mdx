This page is a checklist for building Tables that are easy to run, review, and maintain.

## Table shape

Name the Table after the workflow or decision it supports.

Good names:

- `Support Bot Regression Set`
- `Billing Assistant Prompt Backtest`
- `RAG Ground Truth QA`
- `Refund Policy Request Import`

A useful Table has one clear purpose. If the rows need different inputs, different scoring, or different owners, split the work into another sheet or another Table.

## Row pattern

Each row should be one runnable example.

| Row source | Use it for |
| --- | --- |
| CSV upload | Curated test sets, benchmark rows, manually labeled examples. |
| Request history import | Production or staging traffic, real user prompts, regressions from observability. |
| Manual entry | Small experiments, edge cases, reviewer-created examples. |

For imported request history, filter first. A focused Table is easier to rerun and review than a broad import of unrelated traffic.

## Column pattern

Keep the sheet readable from left to right.

1. **Inputs**: user message, context, expected answer, labels, metadata.
2. **Prompt outputs**: Prompt Template columns, model variants, generated responses.
3. **Transformations**: code, parsing, extraction, normalization.
4. **Checks**: comparisons, assertions, boolean checks, numeric scores.
5. **Review fields**: notes, owner, status, next action.

Computed columns should sit after the columns they depend on.

## Source-mapping pattern

Prompt Template columns should map every template variable to a source column.

When configuring a Prompt Template column:

1. Open **Configure column**.
2. Review the input variables from the prompt.
3. Map each variable to a source column.
4. Leave no required input unmapped before running the column.

<Frame>
  <img src="/images/tables/polished/07-source-mapping.png" alt="Prompt Template source mappings in a Table column configuration panel" />
</Frame>

Use source mappings instead of copying values into prompts manually. Mappings make dependencies explicit and let Tables mark stale cells when inputs change.

## Composition pattern

Use **Composition** when a workflow should reuse another sheet, source column, or intermediate result.

Composition is useful when:

- Multiple sheets need the same upstream output.
- You want to isolate edge-case generation from evaluation.
- You need a reusable workflow step without duplicating columns.
- A downstream column should copy values from a stable source.

## Rerun pattern

Treat amber stale cells as work that needs to be refreshed.

A cell can become stale when you change:

- An input value.
- A Prompt Template configuration.
- A source mapping.
- A dependency column.
- A scoring configuration.

During iteration, rerun the smallest useful scope: one cell, one computed column, or stale cells only. Before using a Table for a release or report, rerun the full workflow that matters.

## Scoring pattern

Configure scoring around the columns that answer the decision you care about.

| Scoring mode | Use it when |
| --- | --- |
| **Auto detect** | Score columns already produce obvious boolean or numeric values. |
| **Boolean** | Checks pass or fail. |
| **Numeric** | Scores are numbers and should be averaged or compared. |
| **Custom code** | The score needs custom aggregation logic. |

Keep score columns separate from reviewer notes. Notes are useful for humans, but score columns should be predictable enough to recalculate.

## History and analytics pattern

Use **History** to compare important changes. Check history after imports, prompt changes, scoring changes, and full reruns.

Use **Analytics** when you need request-level debugging: latency, cost, tokens, model behavior, request status, or trace context for Table-generated work.

<Frame>
  <img src="/images/tables/polished/18-analytics.png" alt="Analytics drawer for request-level debugging from a Table" />
</Frame>

## API pattern

Use the API for repeatable workflows:

1. Create or find the Table.
2. Create or find the sheet.
3. Import CSV data or request logs.
4. Create columns.
5. Add rows if needed.
6. Queue a recalculation operation.
7. Poll the operation status.
8. Read the score, rows, cells, or versions.
