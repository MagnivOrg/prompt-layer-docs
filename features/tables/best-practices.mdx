Use Tables as the primary surface for new evaluation and dataset workflows. A good Table is easy to scan, cheap to rerun, and explicit about what each column represents.

## Start with a clear Table shape

Name the Table after the decision it supports, not the mechanism that created it.

Good names:

- `Support Bot Regression Set`
- `Model Comparison: Billing Assistant`
- `RAG Ground Truth QA`
- `Prod Backtest: Refund Policy Prompt`

Avoid names like `Untitled Table`, `June test`, or `eval v2`. Those do not tell a teammate what the Table proves.

## Organize columns left to right

Put stable inputs on the left and computed outputs on the right.

1. **Input columns**: user message, context, expected answer, labels, metadata, or scenario tags.
2. **Prompt output columns**: outputs from prompt templates, direct LLM calls, or model variants.
3. **Transformation columns**: code, extraction, parsing, or normalization.
4. **Evaluation columns**: comparisons, LLM-as-judge checks, boolean assertions, or numeric scores.
5. **Notes columns**: reviewer comments, triage status, owner, or next action.

This keeps dependencies predictable. If a computed column depends on another computed column, place the dependency earlier in the sheet.

## Use presets when they match the job

Start from a preset when the workflow is already clear. Presets reduce setup work and help teammates understand the intent of the Table.

Use **Blank table** only when you need a custom shape or you are exploring the right workflow.

## Import data where it already lives

Use CSV upload for curated test sets and request-history import for real production or staging traffic.

<Frame>
  <img src="/images/tables/polished/06-upload-menu.jpg" alt="Upload menu for importing CSV files or request history into Tables" />
</Frame>

For request-history imports:

- Filter before importing so the Table is focused.
- Include metadata fields that explain the row, such as environment, route, user segment, or experiment tag.
- Import enough examples to catch regressions, but keep the Table small enough that reruns stay fast.
- Preserve the request or trace context when you need to debug failures later.

  <img src="/images/tables/polished/07-request-history-import.jpg" alt="Request history import dialog for Tables" />
</Frame>

## Keep rows meaningful

Each row should represent one test case, request, scenario, or batch item. Add columns that make the row self-explanatory:

| Column | Why it helps |
| --- | --- |
| `scenario` | Groups rows by user intent or behavior class. |
| `expected_answer` | Makes ground-truth checks explicit. |
| `source_request_id` | Links imported cases back to observability. |
| `severity` | Prioritizes regressions and known edge cases. |
| `owner` | Makes follow-up work actionable. |

## Make scoring explicit

Configure a score for the sheet before using a Table as a release gate or regression signal.

<Frame>
  <img src="/images/tables/polished/08-score-details.jpg" alt="Score details panel for a Table sheet" />
</Frame>

Recommended scoring patterns:

- Use boolean scoring for pass/fail checks, assertions, and exact-match style tests.
- Use numeric scoring when multiple quality dimensions should contribute to an aggregate score.
- Use custom scoring only when the default column-based score cannot represent the decision you need.
- Keep score columns separate from reviewer notes so the score can be recalculated reliably.

## Rerun intentionally

Editing an input, prompt, or computed-column configuration can make dependent cells stale. Recalculate only the rows and columns you need while iterating, then rerun the full sheet before using the result for a release decision.

For API-driven workflows, use Table operations to queue recalculation, poll progress, and cancel long-running work when needed.

## Use versions as checkpoints

A Table sheet version is a checkpoint of the sheet's structure and data. Save or inspect versions around important changes:

- After importing a new dataset or request-history slice.
- Before changing prompt versions or model settings.
- After a large recalculation completes.
- Before using a score in CI, release review, or reporting.

## Use analytics for execution debugging

Tables connect generated cells back to request metrics. Use analytics when you need to understand cost, latency, token usage, model behavior, or request-level failures.

  <img src="/images/tables/polished/10-analytics.jpg" alt="Analytics view opened from a Table" />

## Prefer Tables over legacy Datasets and Reports

For new work, do not create a separate Dataset, Evaluation pipeline, and Report unless you are maintaining an existing legacy workflow. Tables combine those concepts into one versioned surface:

- Table rows replace dataset rows.
- Text columns replace dataset fields.
- Computed columns replace evaluation steps.
- Recalculation replaces report runs.
- Sheet scoring replaces report score cards.
- Sheet versions replace separate dataset and report histories.

See [Migrate from Evaluations and Datasets](/features/tables/migrate-from-evaluations-and-datasets) for a detailed mapping.

## API best practices

When automating Tables through the REST API:

- Create Tables in the right folder with `folder_id` so they are discoverable.
- Treat cursor values as opaque when listing tables, sheets, columns, rows, and operations.
- Use CSV import for prepared data and request-log import for historical traffic.
- Add rows in small batches instead of building one very large request.
- Poll operation endpoints for imports and recalculations before reading final results.
- Store Table, sheet, column, operation, and version IDs in your own system when you need stable automation.
