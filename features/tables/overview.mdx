Tables are PromptLayer's workspace for dataset, evaluation, report, backtesting, and batch workflows. A Table keeps the input data, computed work, score, history, and analytics in one place instead of splitting them across legacy Datasets, Evaluations, and Reports.

A Table has four core objects:

| Object | What it does |
| --- | --- |
| **Table** | The top-level workspace object. Use one Table for a related evaluation or batch workflow. |
| **Sheet** | A tab inside a Table. Each sheet has its own rows, columns, score configuration, history, and analytics scope. |
| **Column** | A field in the sheet. Text columns store input data. Computed columns run prompts, code, checks, extraction, composition, or other work. |
| **Cell** | A row-column value. Text cells are editable inputs. Computed cells have statuses such as queued, running, completed, failed, or stale. |

## Start a Table

A new Table opens with one sheet, one text column, and one row. The empty state gives three starting points:

1. **Create from a prompt**: set up prompt inputs, an output column, and evaluation checks.
2. **Import request history**: select logged requests and add them as rows.
3. **Start blank**: build the sheet manually from rows and columns.

Use **Create from a prompt** when the workflow starts from a Prompt Registry template. Use **Import request history** when production or test traffic is already logged. Use **Start blank** when you want to model the sheet yourself.

## Main workflow

Most Tables follow the same loop:

1. Create or open a Table.
2. Add rows from CSV, request history, or manual entry.
3. Add text columns for inputs, labels, expected answers, or metadata.
4. Add computed columns for prompts, code, assertions, extraction, comparisons, composition, or helper functions.
5. Map each computed column to its source columns.
6. Run the Table, a column, a row, or selected cells.
7. Inspect completed, failed, pending, and stale cells.
8. Configure scoring.
9. Use history and analytics to compare versions and debug requests.

## When to use Tables

Use Tables for:

- Prompt regression tests.
- Batch runs over rows of examples.
- Request-log review and replay.
- Dataset creation from CSV or request history.
- Evaluation pipelines with multiple computed steps.
- Model, prompt, or workflow comparisons.
- Human review queues.
- Scorecards, version history, and analytics for a sheet.

Use the legacy Evaluation and Dataset pages only for existing legacy workflows. For new work, build the workflow in Tables.
